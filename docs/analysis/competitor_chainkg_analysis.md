# ChainKnowledgeGraph 竞品分析报告

本文档从“准备做一个新的金融图谱项目”的视角，对当前项目 `ChainKnowledgeGraph` 做功能梳理、技术分析、优缺点评估和差异化机会分析。

分析依据主要来自本仓库的 `readme.md`、`build_graph.py` 和 `data/` 目录下的数据文件。

## 1. 项目定位

`ChainKnowledgeGraph` 是一个面向 A 股上市公司的产业链知识图谱项目。它试图把上市公司、行业、产品及其上下游关系组织成图数据库，方便用户从产业链视角观察企业、行业和产品之间的关联。

它更像是一个“图谱数据集 + Neo4j 构建脚本”的开源项目，而不是一个完整的金融图谱产品。项目重点在于提供初始数据和建图方式，展示产业链图谱的基本结构。

核心定位可以概括为：

```text
面向 A 股上市公司
以产业链为主题
以 Neo4j 图数据库为承载
覆盖公司、行业、产品三类实体
覆盖所属行业、行业层级、主营产品、产品上下游、产品小类等关系
```

## 2. 功能概览

### 2.1 数据导入功能

项目提供 `build_graph.py`，用于把 `data/` 目录下的 JSON Lines 数据导入 Neo4j。

主要流程：

```text
读取 company.json
读取 product.json
读取 industry.json
批量创建 company/product/industry 节点
为部分 label 创建 name 属性约束或索引
读取关系文件
批量创建公司-行业、行业-行业、公司-产品、产品-产品关系
```

从功能层面看，它完成了图谱构建的最小闭环：

```text
原始结构化数据 -> Neo4j 节点 -> Neo4j 关系 -> Neo4j Browser 可视化查看
```

### 2.2 实体建模功能

项目包含 3 类核心实体：

| 实体类型 | Neo4j Label | 数据文件 | 数量 | 主要属性 |
|---|---|---:|---:|---|
| 公司 | `company` | `company.json` | 4,654 | `name`, `fullname`, `code`, `location`, `time` |
| 行业 | `industry` | `industry.json` | 511 | `code`, `name` |
| 产品 | `product` | `product.json` | 95,559 | `name` |

其中公司实体包含股票代码、上市地点和上市时间，行业实体包含行业代码，产品实体只有名称。

### 2.3 关系建模功能

项目实际导入 4 个关系文件：

| 关系文件 | 起点 | 终点 | 关系类型 | 行数 |
|---|---|---|---|---:|
| `company_industry.json` | 公司 | 行业 | `所属行业` | 4,430 |
| `company_product.json` | 公司 | 产品 | `主营产品` | 53,785 |
| `industry_industry.json` | 行业 | 行业 | `上级行业` | 480 |
| `product_product.json` | 产品 | 产品 | `上游材料` / `下游产品` / `产品小类` | 110,153 |

`product_product.json` 中的关系类型分布：

| 关系类型 | 数量 |
|---|---:|
| `上游材料` | 56,826 |
| `产品小类` | 52,937 |
| `下游产品` | 390 |

项目 README 中提到“6 大类关系”，但从当前脚本实际导入看，关系可归为：

```text
公司 -> 行业：所属行业
行业 -> 行业：上级行业
公司 -> 产品：主营产品
产品 -> 产品：上游材料、下游产品、产品小类
```

### 2.4 Neo4j 可视化查看

项目没有自带前端页面或业务查询系统。用户主要通过 Neo4j Browser 查看图谱。

典型查看方式：

```cypher
MATCH (n) RETURN n LIMIT 50;
MATCH p=(a)-[r]->(b) RETURN p LIMIT 100;
MATCH (c:company)-[:主营产品]->(p:product) RETURN c, p LIMIT 50;
```

因此，它的“界面能力”主要依赖 Neo4j Browser 的默认图可视化，而不是项目自身提供的产品界面。

## 3. 数据结构分析

### 3.1 公司数据

样例：

```json
{
  "name": "邵阳液压",
  "fullname": "邵阳维克液压股份有限公司",
  "code": "301079",
  "location": "深圳证券交易所",
  "time": "2021-10-19"
}
```

特点：

```text
有上市公司简称
有公司全称
有股票代码
有上市地点
有上市时间
```

不足：

```text
没有交易所标准后缀，例如 SZ/SH/BJ 不统一
没有行业分类以外的财务指标
没有市值、营收、利润、PE、PB、ROE 等金融字段
没有公司状态、退市状态、上市板块等结构化字段
公司 name 存在重复
```

### 3.2 行业数据

样例：

```json
{
  "code": "110000",
  "name": "农林牧渔"
}
```

特点：

```text
行业节点有 code 和 name
行业之间有上级行业关系
适合表达申万行业树或类似层级结构
```

不足：

```text
没有明确写入行业层级，例如一级、二级、三级
没有行业标准来源字段
没有行业生效日期或版本字段
没有行业指数、估值、景气度等金融属性
```

### 3.3 产品数据

样例：

```json
{
  "name": "照明系统安装"
}
```

特点：

```text
产品数量大，覆盖面广
适合做产业链上下游扩展
```

不足：

```text
产品实体只有 name，没有 code、类别、来源、置信度
产品命名粒度不稳定，可能混合产品、服务、工艺、材料、业务描述
缺少标准化体系，后续实体对齐难度较高
```

### 3.4 公司-产品关系

样例：

```json
{
  "company_code": "688268.SH",
  "product_name": "特种气体",
  "rel_weight": 0.510725,
  "rel": "主营产品",
  "company_name": "华特气体"
}
```

特点：

```text
包含主营产品关系
部分关系有 rel_weight，可用于衡量主营产品重要性
是项目最有金融图谱价值的一类关系
```

数据质量观察：

```text
company_product.json 共 53,785 行
其中 rel_weight 数值字段有 39,072 行
rel_weight 最小值约 -59.02，最大值约 3.20，均值约 0.217
存在负数、0、超过 1 的值，需要解释和清洗
```

这说明 `rel_weight` 可能不是严格意义上的营收占比，或者数据抽取后没有充分归一化。若用于投资分析，需要谨慎。

## 4. 技术实现分析

### 4.1 技术栈

项目使用：

```text
Python
py2neo
Neo4j
JSON Lines 文件
Cypher
```

整体架构非常轻：

```text
data/*.json -> build_graph.py -> py2neo -> Neo4j
```

优点是易理解、易运行；缺点是工程化程度较低。

### 4.2 节点创建逻辑

代码使用 `UNWIND` 批量创建节点：

```cypher
UNWIND $batch AS row
CREATE (n:label)
SET n = row
```

优点：

```text
比逐条创建快
实现简单
直接保留原始属性
```

问题：

```text
使用 CREATE，不是 MERGE，重复运行会重复建点
没有主键去重逻辑
没有批次大小控制，超大数据时可能有内存压力
没有事务失败后的恢复机制
```

### 4.3 关系创建逻辑

代码按 `rel` 字段分组，然后动态创建关系类型：

```cypher
UNWIND $batch AS row
MATCH (a:start_label {name: row.from})
MATCH (b:end_label {name: row.to})
CREATE (a)-[rel:`关系类型`]->(b)
```

优点：

```text
可以支持中文关系类型
不同关系类型可以自动分组
实现方式灵活
```

问题：

```text
关系匹配依赖 name，不依赖 code
如果 name 重复，会产生多重匹配和多余关系
如果起点或终点名称缺失，关系会静默丢失
关系使用 CREATE，重复运行会重复建边
关系没有唯一约束或去重策略
```

### 4.4 索引和约束逻辑

项目尝试为以下 label 创建 `name` 唯一约束：

```text
company.name
product.name
industry.name
```

但数据中公司简称存在重复：

```text
company.json 共 4,654 行
唯一 company.name 为 4,605 个
重复公司名种类 33 个
重复行数 49 行
```

因此 `company.name` 唯一约束无法可靠创建。

典型重复公司名包括：

```text
龙软科技
映翰通
奥赛康
京源环保
恒林股份
风语筑
```

这会影响后续图谱质量：如果关系按 `company.name` 匹配，重复公司名会导致关系被错误扩增。

## 5. 当前项目的功能边界

这个项目已经具备：

```text
基础产业链图谱数据
公司、行业、产品三类实体
产品上下游关系
公司主营产品关系
公司所属行业关系
行业上下级关系
Neo4j 导入脚本
Neo4j Browser 查看能力
```

但它不具备：

```text
完整 Web 应用
业务 API
自然语言问答
金融指标查询
定时更新
增量更新
数据血缘管理
数据质量看板
实体消歧系统
权限系统
多源数据融合
投研工作流
风险传导分析引擎
```

因此它更像一个“开放数据底座”或“图谱样例工程”，不是完整的金融图谱产品。

## 6. 优点分析

### 6.1 选题切入点好

产业链图谱对金融场景有天然价值：

```text
理解上市公司主营业务
分析公司上下游暴露
观察行业传导关系
支持主题投资和产业链研究
辅助寻找相似公司和替代标的
```

相比普通企业知识图谱，产业链图谱更接近投研需求。

### 6.2 数据规模较大

项目包含：

```text
4,654 家上市公司
511 个行业
95,559 个产品
约 16.9 万条关系
```

对于课程项目、原型系统或入门研究来说，规模已经足够支撑可视化展示和基础分析。

### 6.3 Schema 简单清晰

三类实体：

```text
公司
行业
产品
```

几类关系：

```text
所属行业
上级行业
主营产品
上游材料
下游产品
产品小类
```

这种模型非常容易理解，也容易迁移到其他图数据库或业务系统。

### 6.4 Neo4j 表达自然

产业链上下游、行业层级、公司主营产品非常适合图数据库表达。

Neo4j 的优势包括：

```text
关系可视化直观
Cypher 查询路径方便
适合做多跳关联
适合探索式分析
```

### 6.5 数据文件开放透明

项目没有把数据藏在数据库 dump 中，而是放在 `data/*.json` 文件里。这样便于：

```text
检查数据结构
二次清洗
迁移到其他数据库
重新设计 schema
做训练集或评测集
```

### 6.6 对新项目有参考价值

它提供了一个可复用的起点：

```text
金融产业链图谱可以有哪些实体
上市公司和产品如何连接
行业层级如何表达
产品上下游如何表达
Neo4j 中如何组织中文关系
```

## 7. 缺点分析

### 7.1 工程化程度较低

项目只有一个主要构建脚本，缺少完整工程结构。

缺失内容包括：

```text
requirements.txt 或 pyproject.toml
自动化安装脚本
配置文件
日志系统
测试用例
数据校验流程
错误重试机制
增量导入机制
```

这会导致用户拿到项目后容易遇到环境问题、版本问题和重复导入问题。

### 7.2 脚本不可安全重复运行

节点和关系都使用 `CREATE`。重复运行会重复插入数据。

实际表现：

```text
重复运行后 company 节点可能翻倍
product 节点可能因唯一约束报错
关系可能重复创建
```

这对真实项目是严重问题。一个生产级图谱系统至少需要：

```text
唯一主键
MERGE 或批量 upsert
导入批次标识
重复导入检测
清库或重建命令
```

### 7.3 公司实体消歧不足

公司关系使用 `company_name` 匹配，而不是稳定的 `company_code`。

问题：

```text
公司简称可能重复
公司名称可能变更
ST、退市、简称变更会造成匹配不稳定
不同市场可能存在同名或近似名
```

更合理的方式是以证券代码作为主键：

```text
company.code
```

并把简称、全称作为可变属性。

### 7.4 产品实体标准化不足

产品只有 `name` 字段，没有类别、代码、来源、置信度或标准词表映射。

这会导致：

```text
同义词难以合并
上下位关系难以验证
产品、服务、材料、工艺容易混杂
粒度不稳定
后续搜索和推荐质量受限
```

如果要做更强的金融图谱，产品层需要单独建设标准化体系。

### 7.5 关系置信度和来源不足

关系文件中基本没有：

```text
数据来源
抽取方法
置信度
证据文本
更新时间
生效时间
失效时间
```

这会限制金融场景使用。投研和风控场景通常需要知道：

```text
这条关系从哪里来
是不是最新
可信度多高
有没有原文证据
什么时候生效
是否已经过期
```

### 7.6 金融属性不足

项目虽然定位在金融领域，但金融指标本身较少。

缺少：

```text
行情数据
市值
估值指标
财务报表
主营收入构成
毛利率
股东信息
公告事件
研报事件
新闻舆情
风险事件
供应商/客户数据
地域产能
```

因此它更偏“产业链数据图谱”，不是“金融分析图谱”。

### 7.7 没有应用层

用户只能通过 Neo4j Browser 查询。对非技术用户不友好。

缺少：

```text
公司详情页
行业详情页
产品详情页
路径探索界面
筛选器
图谱搜索
风险传播分析
导出功能
接口服务
权限和账号体系
```

这给新项目留下了很大的产品化空间。

### 7.8 没有更新机制

产业链和上市公司信息是动态变化的，但项目没有提供更新策略。

真实金融项目至少需要考虑：

```text
新上市公司加入
退市公司处理
公司改名
行业分类调整
主营业务变化
产品上下游变化
定期公告更新
数据版本回溯
```

### 7.9 数据质量问题会影响图谱可信度

本地检查发现：

```text
company.json 中 company.name 有重复
company_industry.json 中部分公司名或行业名无法匹配到节点
company_product.json 中部分 product_name 无法匹配到产品节点
product_product.json 中少量产品名无法匹配到产品节点
rel_weight 存在负数和大于 1 的值
```

这些问题并不意味着项目不可用，但说明它更适合作为研究/演示数据，而不是直接用于严肃投研决策。

## 8. 对你的金融图谱项目的启发

### 8.1 可以借鉴的部分

建议借鉴：

```text
以图数据库表达产业链关系
公司、行业、产品作为基础三元骨架
公司主营产品关系作为核心入口
产品上下游作为产业链扩展层
行业层级作为导航和聚合维度
Neo4j Browser 或图可视化作为早期验证工具
```

这些设计简单、直观、容易出 Demo。

### 8.2 应该避免的问题

建议避免：

```text
用名称作为主键
脚本重复运行会重复建点
没有数据版本
没有关系来源
没有关系置信度
没有清洗和校验报告
只提供 Neo4j Browser，不提供业务界面
只做静态数据导入，不做更新流程
```

这些是当前竞品最明显的工程短板。

### 8.3 你的项目可以做出的差异化

如果你也做金融图谱，可以从以下方向超过它。

#### 方向一：更金融化

在产业链基础上增加金融属性：

```text
公司行情
市值
估值
财务指标
主营收入构成
利润贡献
公告事件
研报观点
新闻舆情
风险事件
股东和实控人
供应商和客户
```

这样项目就不只是产业链图谱，而是能服务投研分析的金融知识图谱。

#### 方向二：更可解释

给每条关系增加：

```text
source 数据来源
evidence 证据文本或链接
confidence 置信度
extract_method 抽取方法
update_time 更新时间
valid_from 生效时间
valid_to 失效时间
```

这样用户可以判断关系是否可信。

#### 方向三：更工程化

建设完整数据管线：

```text
原始数据采集
清洗标准化
实体消歧
关系抽取
质量检查
版本管理
增量更新
Neo4j 导入
API 服务
前端可视化
```

这会明显区别于当前竞品的一次性导入脚本。

#### 方向四：更产品化

做面向业务用户的功能：

```text
公司画像页
行业画像页
产品画像页
产业链路径探索
上下游风险传导
相似公司推荐
主题投资图谱
供应链暴露分析
事件影响范围分析
```

这些能力比单纯 Neo4j Browser 更接近真实用户需求。

#### 方向五：更适合 LLM/RAG

如果你的项目要结合大模型，可以把图谱变成检索和推理底座：

```text
实体识别和链接
自然语言转 Cypher
图谱增强问答
多跳路径解释
关系证据召回
投研报告生成
风险链条解释
```

当前竞品没有 LLM 层，这是一个明显机会。

## 9. 建议的新项目 Schema

如果你的目标是做一个更完整的金融图谱，可以考虑以下实体：

```text
Company        上市公司
Security       证券
Industry       行业
Product        产品/业务
Concept        题材/概念
Person         高管/实控人
Shareholder    股东
Report         公告/研报/财报
Event          新闻/风险事件
Metric         财务或行情指标
Region         地区
Institution    机构
```

建议关系：

```text
Company -[:LISTED_AS]-> Security
Company -[:BELONGS_TO_INDUSTRY]-> Industry
Industry -[:PARENT_INDUSTRY]-> Industry
Company -[:HAS_PRODUCT {revenue_ratio, source, confidence}]-> Product
Product -[:UPSTREAM_OF]-> Product
Product -[:DOWNSTREAM_OF]-> Product
Company -[:SUPPLIES_TO]-> Company
Company -[:CUSTOMER_OF]-> Company
Company -[:HAS_CONCEPT]-> Concept
Company -[:HAS_SHAREHOLDER]-> Shareholder
Person -[:MANAGES]-> Company
Report -[:MENTIONS]-> Company
Event -[:AFFECTS]-> Company
Metric -[:MEASURES]-> Company
```

关键建议：

```text
公司主键用股票代码或统一 company_id，不用简称
行业主键用行业标准 + 行业代码 + 版本
产品需要标准词表和别名表
关系必须有来源和更新时间
导入脚本必须可重复运行
```

## 10. 建议的 MVP 路线

### 阶段一：复刻并修正竞品

目标：做出一个更干净的产业链图谱底座。

任务：

```text
整理上市公司基础信息
整理行业层级
整理主营产品
整理产品上下游
使用 code/id 作为主键
使用 MERGE 实现幂等导入
增加数据质量检查报告
```

验收标准：

```text
可以重复运行导入脚本
节点数稳定
关系数稳定
无明显孤立关系
每类实体有唯一约束
```

### 阶段二：做出金融分析能力

目标：超过“静态产业链图谱”。

任务：

```text
接入行情或财务指标
接入公告/新闻/研报事件
构建公司画像
构建行业画像
构建产品画像
支持上下游路径查询
支持主题/概念查询
```

验收标准：

```text
用户可以回答“某公司受哪些上游原材料影响”
用户可以回答“某产品涨价影响哪些公司”
用户可以回答“某行业有哪些核心上市公司和主营产品”
```

### 阶段三：做产品化界面

目标：让非 Neo4j 用户也能使用。

任务：

```text
搜索公司/行业/产品
展示详情页
展示图谱路径
提供筛选器
提供导出能力
提供 API
```

验收标准：

```text
不懂 Cypher 的用户也能完成核心分析
图谱查询结果有解释和证据
可以沉淀常用分析模板
```

### 阶段四：结合 LLM

目标：形成“金融图谱 + 大模型”的智能分析系统。

任务：

```text
自然语言问答
自然语言转 Cypher
图谱路径解释
证据引用
投研摘要生成
风险链条推理
```

验收标准：

```text
回答包含数据来源
回答能解释路径
回答能区分事实、推断和不确定性
```

## 11. SWOT 总结

### Strengths 优势

```text
主题有价值，切中产业链金融分析需求
数据规模较大，适合展示和研究
Schema 简洁，学习成本低
Neo4j 可视化效果直观
数据文件开放，方便二次开发
```

### Weaknesses 劣势

```text
工程化弱
不可安全重复导入
主键设计不足
公司和产品消歧不足
缺少数据来源和置信度
缺少金融指标
缺少应用层
缺少更新机制
```

### Opportunities 机会

```text
金融图谱产品化空间大
可以结合公告、新闻、研报和财务数据
可以结合 LLM 做智能问答和路径解释
可以服务投研、风控、产业研究、主题投资
可以做成比 Neo4j Browser 更友好的分析工具
```

### Threats 威胁

```text
产业链数据主观性强，标准难统一
产品实体标准化成本高
高质量供应链/客户数据获取难
金融数据更新频率高，维护成本大
如果缺少可解释证据，用户信任度会低
```

## 12. 对新项目的结论建议

如果你的目标是做一个类似但更强的金融图谱项目，不建议只复刻当前项目。更好的路线是：

```text
以当前项目作为产业链骨架参考
用更稳定的主键和更严格的数据质量体系重建底座
补充金融指标、公告事件、新闻舆情和证据来源
建设可重复运行的数据管线
提供面向业务用户的图谱应用界面
再叠加 LLM 问答和路径解释能力
```

一句话总结：

```text
ChainKnowledgeGraph 的价值在于给出了一个产业链图谱原型；它的短板在于还没有形成可靠、可更新、可解释、可产品化的金融图谱系统。
```
